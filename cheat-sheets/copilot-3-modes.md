## ⚡ Quick Decision

| Situation | Mode | Why |
|---|---|---|
| Understand code, discuss design, plan approach | **Chat** | Conversation, low cost, reversible |
| Create/edit multiple related files at once | **Edits** | Multi-file view, you review diff |
| Delegate complete task (issue to PR) | **Agent** | Works alone, you review at the end |

---

## 💬 Chat — Conversation

**Use when**: you don't know what you want; want to understand; want to discuss; want to evaluate trade-offs.

**Phrases that work:**
- "Explain what this Natural program does line by line."
- "What risks does using `JSONB` to store bank account history have?"
- "Summarize this DDM in 5 lines for someone who doesn't know Adabas."
- "Challenge the following ADR: `{paste ADR}`."

**Common mistakes:**
- Using Chat to generate a file. Use Edits instead.
- Accepting answers without validation. Copilot hallucinates — always verify.
- Prompt too short ("help"). Give context: what you have, what you want, what you've tried.

---

## ✏️ Edits — Batch Editing

**Use when**: you know what you want; you want changes in multiple files; you have structured context.

**Phrases that work:**
- "Create the `beneficiary`, `agreement`, `payment` modules with standard Spring Boot package structure."
- "Add unit test for each public method in `PaymentService`."
- "Rename `Convenio` to `Agreement` across the project and update references."
- "For each existing Flyway migration, add a commented-out rollback."

**Common mistakes:**
- Scope too large. Break it into steps.
- Not reviewing the diff. Always look before accepting.
- Mixing logical changes with renaming. One PR per purpose.

---

## 🕹️ Agent — Autonomous Delegation

**Use when**: you have a well-described issue, you accept that it will take time, you're willing to review PR generated by someone other than you.

**How to prepare**:
1. Write the issue with **context, acceptance criteria, scope**.
2. Point to relevant files ("read `docs/adr/001.md` before starting").
3. State what NOT to do ("don't alter the PostgreSQL schema").

**Monitoring**: don't operate while the Agent runs. Let it go. Check every ~10 min if the path makes sense.

**Reviewing the Agent's PR**: exactly like reviewing a human PR. Express review is still review.

**Common mistakes:**
- Vague issue leads to garbage output.
- Starting Agent on a 5-minute task that Edits would solve.
- Merging without review because "it's just the Agent".

---

## 🎯 Rule of Thumb

> If you didn't know it was AI, would you accept this code in your project? If not, reject or refine. Copilot accelerates those who know; it doesn't replace judgment.
